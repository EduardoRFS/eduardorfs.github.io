Functional programming and problem space

Any kind of problem that you have at your work can be described as “someone forgot about an edge case”. The best way to mitigate that is having a smaller problem space and FP reduces the problem space compared to impure OOP in mostly two ways specificity and locality.

The main feature of OOP is depending on interfaces instead of implementation by default. This makes code more abstract, you literally cannot predict the behaviour of a function such as x => x.add(x), on the other hand the function x => x + x can do only a few things.

Impurity violates abstractions, they allow you to have implicit back channels, in the general case to understand the behaviour of a piece of mutable code you may need to understand the entire program.

Static typing takes this even further, by making invalid states not even representable but it is not an FP specific thing.
